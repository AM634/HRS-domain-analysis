# 演習3：システム分析

## 1. 目的

演習2で作成したユースケースをもとに、HRSのシステム分析を行う。
各ユースケースについて、バウンダリ、コントロール、エンティティを識別し、UMLコラボレーション図として表現する。
また、それらの分析結果をもとに、システム分析クラス図を作成する。

## 2. 対象ユースケース

本演習では、以下の3つのユースケースを対象とする。

1. 空室を確認して部屋を予約する
2. 予約番号でチェックインする
3. 宿泊料を精算してチェックアウトする

## 3. コラボレーション図

### 3.1 空室を確認して部屋を予約する

PlantUMLファイル：

[予約コラボレーション図](../diagrams/collaboration_reservation.puml)

この図では、利用者が予約画面に宿泊条件を入力し、予約処理が部屋情報を検索して空室一覧を返す流れを表した。
その後、利用者が部屋を選択すると、予約処理が予約を作成し、予約番号を返す。

### 3.2 予約番号でチェックインする

PlantUMLファイル：

[チェックインコラボレーション図](../diagrams/collaboration_checkin.puml)

この図では、利用者がホテル受付係に予約番号を伝え、受付係がチェックイン画面に予約番号を入力する流れを表した。
チェックイン処理は予約情報と部屋情報を確認し、予約状態をチェックイン済みに更新する。

### 3.3 宿泊料を精算してチェックアウトする

PlantUMLファイル：

[チェックアウトコラボレーション図](../diagrams/collaboration_checkout.puml)

この図では、利用者がチェックアウトを申し出た後、受付係が部屋番号または予約番号を入力し、チェックアウト処理が宿泊情報を検索する流れを表した。
その後、宿泊料を計算し、支払い完了後に宿泊状態と部屋状態を更新する。

## 4. システム分析クラス図

PlantUMLファイル：

[システム分析クラス図](../diagrams/analysis_class_diagram.puml)

システム分析クラス図では、以下の3種類のクラスを区別した。

| 種類 | 役割 | 例 |
|---|---|---|
| バウンダリ | アクタとシステムのやり取りを担当する | 予約画面、チェックイン画面、チェックアウト画面 |
| コントロール | ユースケースごとの処理の流れを制御する | 予約処理、チェックイン処理、チェックアウト処理 |
| エンティティ | システム内で扱うデータや概念を表す | 予約、部屋、宿泊、宿泊料、支払い |

## 5. 演習1・演習2からの修正点

演習1のドメイン分析では、HRSの主要概念として利用者、予約、部屋、宿泊、宿泊料、支払いなどを抽出した。
演習3では、それらに加えて、ユースケースの処理を表すコントロールクラスと、アクタとのやり取りを表すバウンダリクラスを追加した。

また、演習2のユースケース記述に対応するように、各コラボレーション図のメッセージ順序を整理した。

## 6. 工夫点

ユースケースごとにコントロールクラスを1つずつ用意した。
これにより、予約、チェックイン、チェックアウトの処理が分離され、どのユースケースの処理か分かりやすくなった。

また、利用者が直接HRSを操作する場面と、ホテル受付係を通してHRSを操作する場面を区別した。
これにより、HRSの実際の利用場面に近い分析モデルになった。
